## 3. Dispatch Logic (inside `dispatchSTOHandler`)

**Per STO line:**
1. Check stock availability: SELECT stock_snapshot WHERE company_id=sending_company, material_id=line.material_id, stock_type_code='UNRESTRICTED'. Qty must be >= line.quantity. If not â†’ 400 INSUFFICIENT_STOCK.
2. Call `post_stock_movement()`:
   - p_movement_type_code = 'STO_ISSUE' (OUT)
   - p_direction = 'OUT'
   - p_stock_type_code = 'UNRESTRICTED'
   - p_unit_value = current snapshot valuation_rate
3. Update STO line: issued_qty += line.quantity, balance_qty = quantity - issued_qty
4. Store stock_document_id on STO line

**After all lines:**
5. INSERT delivery_challan (dc_type='STO', sto_id=this STO id, company_id=sending_company)
6. INSERT delivery_challan_line for each STO line
7. INSERT gate_exit_outbound (exit_type='STO', sto_id, dc_id)
8. STO status â†’ DISPATCHED

## 4. Business Rules

| Rule | Detail |
|---|---|
| Both companies see STO | listSTOsHandler filters: WHERE sending_company_id = :company OR receiving_company_id = :company |
| DC auto-generated | Delivery Challan is NEVER manually created for STO. Auto-generated by dispatchSTOHandler. |
| Gate Exit auto-generated | Similarly auto-generated on dispatch. |
| BULK/TANKER weighment | Gate Exit outbound: gross_weight entered at dispatch time. Tare entered after. net_weight = gross - tare. |
| STO receipt = GRN | Receiving company creates GRN via regular Gate Entry + GRN flow with STO reference. |
| balance_qty tracking | Partial dispatch possible (Phase-2). In Phase-1: full dispatch assumed. |
| STO_ISSUE movement | Must exist in movement_type_master (seeded in Gate-11). |
| Transfer price | Last-used pattern: DEFAULT from last STO for this material + company pair. Manual override allowed. |
